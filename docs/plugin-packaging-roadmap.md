# DigestPilot Plugin Packaging Roadmap

DigestPilot should remain platform-neutral at the core. Plugin packaging is a distribution layer for agent runtimes, not a fork of the Skill or CLI contracts.

## Naming

- Product display name: `DigestPilot`
- Plugin id: `digestpilot`
- Repository name target: `digestpilot`
- Existing CLI name: keep `subscription-research`
- Existing npm package target: keep `@subscription-research/cli`
- Existing Skill directory names: keep stable for now

## Packaging Targets

### Claude Code

Target shape:

```text
plugins/digestpilot/
  .claude-plugin/plugin.json
  skills/
    rss-ai-digest/
    public-trend-radar/
    rss-source-curator/
    subscription-research-agent/
  README.md
```

The Claude plugin should install the Skills as a suite. Slash commands can be added later for common workflows such as daily digest, public trend scan, source review, and research brief preparation.

### Codex

Target shape:

```text
plugins/digestpilot/
  .codex-plugin/plugin.json
  skills/
    rss-ai-digest/
    public-trend-radar/
    rss-source-curator/
    subscription-research-agent/
  README.md
.agents/plugins/marketplace.json
```

The Codex plugin should use the same Skill files as the Claude plugin. Marketplace metadata should point to the local plugin package during development and can later point to a published repository.

## Build Rule

The repository root `skills/` directory remains the source of truth. Plugin package `skills/` directories should be generated by a build script, not edited independently.

Planned helper commands:

```bash
node scripts/build-plugin.mjs --target claude
node scripts/build-plugin.mjs --target codex
node scripts/build-plugin.mjs --target all
```

## CLI Distribution

The deterministic runtime stays in `packages/research-cli` and remains callable as `subscription-research`.

Preferred install path after packaging:

```bash
npx @subscription-research/cli ...
```

Until the npm package is published, plugin guidance should document local development usage from the repository checkout.

## First Plugin Scope

The first plugin release should include only:

- Four existing Skills.
- Plugin manifests for Claude Code and Codex.
- Minimal README with install and smoke-test instructions.
- Build script that copies root Skills into `plugins/digestpilot/skills/`.
- Validators for Skill and plugin manifests.

Defer:

- MCP servers.
- Runtime hooks.
- Notification integrations.
- Publisher adapters.
- Separate per-Skill plugin packages.

## Acceptance Criteria

- `skills/` remains the source of truth.
- Generated plugin package validates for the target runtime.
- `DigestPilot` appears as the product/plugin display name.
- Existing Skill names still work.
- Existing `subscription-research` CLI commands still work.
- Public README explains the product as information-stream digest and research evidence tooling, not as an RSS-only Skill repository.
